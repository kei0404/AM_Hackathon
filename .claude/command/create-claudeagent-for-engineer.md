あなたはエンジニアです。以下のひな型をベースにしてClaude Code の実行ルールとなる `CLAUDE.md` ファイルを作成してください。

## CLAUDE.md の形式

```
実装作業に必要な設計を行なって、方針を示してください。
タスクは非常に細かく分割し、一度の指示でエラーなく実装できるような粒度にして下さい。
設計方針と作業のステップを示してください。
CLAUDE.md には以下の8つの要素を記載してください。

1. **メタデータブロック(プロジェクト情報)**

- プロジェクト名
- バージョン番号
- ライセンス情報（指定された場合）
- 作成日/更新日
- 概要: Guidelines

2. **いつPlansを参照するかの規定**

- 以下の文言を必ず入力すること
- PALNS.md が唯一の決定事項の所在とする

---
複雑な機能や大規模なリファクタリングを実装する際は、設計から実装まで必ず ExecPlan（./PLANS.md に記載）を使用してください。

「exec plan」という言葉を使ったときは、 plans.md を参照し、以下を実行してください：

全体像を理解する
進捗状況を確認する
作業後に plans.md を更新する
発見事項と決定事項を記録する
plans.md はあなたの長期記憶であり、 プロジェクトの羅針盤です。

（閾値の明文化）

実装や調査が 2 ステップ以上になりそうなとき、または 30 分を超える見込みの作業では PLANS.md を参照してください。ファイルが無ければ作成し、目的・手順・担当を明記してください。
ステークホルダーが 2 名以上関与する場合や、ロールアウトを段階化する必要がある場合も作成義務があります。
単純なコピー修正や 1 ファイル修正は免除対象ですが、判断に迷ったら作るのが原則です。
---

3. **プロジェクトの目的/背景: Purpose/Context**
- ユーザーがこの変更で何を得るか
- 得たものをどのように確認できるか

4. **コア機能仕様: CoreFunctionalitySpecifications**
- 技術的な詳細を含む主要機能の説明
- 抽象度の高い技術説明を心がける

5. **前提条件/ルール: Top-Level Rules**
- 必ず守られるべきルールを記載する

6. **プロンプト設計指針:Prompt Design Guidelines**
AIとの対話を最適化するための指示を記載する
- システム指示の記載方法
- ユーザー入力の想定パターン
- サンプルI/O フォーマット

7. **注意点/制約/禁止事項: Notification/Constraints/DONT'S**
プロジェクトでの注意点や禁止トピックおよび出力のスタイル等を規定します

8. **更新履歴: Changelog**
バージョン管理のベストプラクティス（Semantic Versioning推奨）に従った変更履歴の記録。

```

## CLAUDE.mdを作成する際に考慮する点
- 指示はできるだけ少なくする
- 汎用的な指示/ルールを心がける
- 指示が長くなりすぎたら、別のファイルに分ける（**段階的開示**）
